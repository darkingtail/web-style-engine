---
title: antdv-style migration
description: Migration scaffold for validating the abstraction against antdv-style.
---

`web-style-engine` should not know about antdv. The migration path keeps antdv-specific behavior inside `antdv-style` or the explicit `web-style-engine/migration` boundary:

- antdv-next `theme.useToken`
- `prefixCls` and `iconPrefixCls`
- antdv token mapping
- antdv CSS variable prefix
- antdv-specific ThemeProvider defaults

The first migration step can wrap the existing Emotion implementation with `createStyleEngineFromEmotion`, then gradually extract `createStylesCore`, split ThemeProvider, integrate responsive helpers, and validate Vue / React / Solid against the same core.

## Boundary Helpers

```ts
import {
  applyAntdvThemeToEngine,
  createAntdvTokenAdapter,
  validateAntdvStyleCompatibility,
} from 'web-style-engine/migration'
```

`createAntdvTokenAdapter()` maps antdv-specific token concerns without moving them into core:

```ts
const adapter = createAntdvTokenAdapter(seedToken, {
  prefixCls: 'ant',
  iconPrefixCls: 'anticon',
  cssVarPrefix: 'ant',
})
```

`applyAntdvThemeToEngine()` injects mapped tokens through CSS variables:

```ts
applyAntdvThemeToEngine(engine, seedToken, {
  scope: '.ant-app',
  prefixCls: 'ant',
})
```

## Compatibility Gate

```ts
validateAntdvStyleCompatibility({
  publicApiStable: true,
  ssrSafe: true,
  hydrationSafe: true,
  multiInstanceSafe: true,
  nestedThemeSafe: true,
  responsivePathDocumented: true,
})
```

The gate is intentionally explicit so `antdv-style` can track migration readiness without coupling `web-style-engine` core to antdv runtime details.
